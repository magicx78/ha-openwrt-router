# v1.19 Scope — Coordinator Stabilization

**Status: PLANNING** — geöffnet 2026-04-30 nach v1.17.9-Crash-Forensik (Production-Host 10.10.10.165, Crash-Kette Log-Viewer-Watchdog → Supervisor-Selbst-Halt).

## Anlass

Die 22h-Sampler-Daten aus `diagnostics/prod-24h-baseline.jsonl` (v1.17.9) zeigen, dass die ursprüngliche FD-Leak-Hypothese **nicht** der primäre Auslöser ist. Stattdessen:

| Beobachtung | Befund |
|---|---|
| `fd_sock` Verlauf | oszilliert um ~125 — KEIN monotones Wachstum |
| `fd_pipe` / `fd_file` | flat (16 / 21) |
| `subproc` | konstant 1 — KEIN Subprocess-Storm |
| RSS | +140 MB / 22h = 6.4 MB/h (moderat) |
| `threads` | +20 / 22h = 0.9 Thread/h (verdächtig) |
| **Coordinator-Update-Dauer** (HA-Log post-Recovery) | **62–65 s pro Router**, bei `SCAN_INTERVAL_SECONDS = 60` → Updates **überlappen** |
| **rpcd-Session-Lifecycle** (HA-Log post-Recovery) | 7+ "Session expired, attempting re-login" innerhalb von ~2 s — Token-TTL läuft mid-update aus |

Der akute Crash am 2026-04-30 wurde durch das **Log Viewer Add-on** ausgelöst, das zyklisch crasht und den Supervisor-Watchdog ins 10-Calls-in-30min-Rate-Limit treibt. Trigger ist behoben (Watchdog disabled, Boot manual). Aber die **Coordinator-Symptome bleiben** und sind die wahrscheinliche Ursache der vorigen sporadischen Crashes.

v1.18.0-Hardening (F4 / F3) adressiert diese Symptome NICHT direkt — der Refactor ist trotzdem korrekt und produktiv (kein Rollback), aber er ist nicht der erwartete Crash-Fix.

## Scope (priorisiert)

### 1. Proactive Token-Refresh vor Ablauf

**Problem.** rpcd-Session-Token TTL ist 300 s (`SESSION_LIFETIME_SECONDS`). Aktueller Code refreshed reaktiv — bei `error_code=6` kommt ein Re-Login. Bei 60-s-Coordinator-Updates und Token-Ablauf mid-update entstehen 4–7 Re-Logins pro Sekunde. Das ist Auth-Hammer auf rpcd, mit zusätzlicher Latenz pro Re-Login.

**Plan.**
- `_call()` prüft VOR jedem RPC-Aufruf `now > self._token_expires_at - SESSION_REFRESH_MARGIN_SECONDS` (Konstante existiert bereits, wird aber nicht durchgängig gehonoriert).
- Refresh wird einmal asynchron triggered, neue Calls warten auf das Refresh-Future (`asyncio.shield`).
- Verhindert dass mehrere parallele `_call()` jeweils ihren eigenen Re-Login starten.
- **Test.** Mock einen Token mit Expires-in-1s, parallele 10 `_call()` → max. 1 `session/login` Aufruf.

**Files.** `custom_components/openwrt_router/api.py` (`_call`, `_login`).

**Risiko.** Niedrig — kompatibel mit bestehendem Code-Pfad.

### 2. Per-Router Coordinator-Lock / Overlap-Schutz

**Problem.** `DataUpdateCoordinator` startet einen neuen `_async_update_data()` wenn das Intervall abläuft, AUCH wenn der vorige Run noch nicht fertig ist. Bei 65-s-Updates und 60-s-Intervall stauen sich Tasks im executor.

**Plan.**
- `OpenWrtCoordinator._async_update_data()` wrapped in `async with self._update_lock:` — pro Coordinator-Instanz.
- Wenn der Lock besetzt ist: aktueller Update SKIP'd (return cached data) und WARNING geloggt.
- HA-DataUpdateCoordinator ruft seinerseits keinen parallelen Update an, aber explicit `async_request_refresh()` aus Buttons / Service-Calls könnte das.
- **Test.** Simuliere Update der 90 s dauert → 2. Update gleitet drüber, returnt sofort, log enthält "skipped".

**Files.** `custom_components/openwrt_router/coordinator.py`.

**Risiko.** Mittel — Veränderung der Update-Semantik. Migration-Hinweis im Changelog.

### 3. Update-Timeout < Scan-Interval

**Problem.** Heute kann ein Update unbegrenzt laufen (oder bis Network/HTTP-Timeout). Bei degraded routers staut sich das.

**Plan.**
- `_async_update_data()` läuft mit `asyncio.wait_for(timeout=SCAN_INTERVAL_SECONDS - 5)`.
- TimeoutError → cached data + WARNING → nächster Cycle versucht es neu.
- Default `SCAN_INTERVAL_SECONDS = 90` (heute 60) — gibt mehr Puffer für slow routers, weniger Overlap-Risiko.
- **Test.** Mock einen `_call()` der 100 s blockiert → Coordinator returnt nach ~85 s mit cached data, kein Hänger.

**Files.** `custom_components/openwrt_router/coordinator.py`, `const.py` (Default).

**Risiko.** Niedrig — Default-Anpassung kann via Config-Flow Override für Test-Setups bleiben (siehe Punkt 5).

### 4. Session-Storm Backoff

**Problem.** Falls ein Router `session/login` rate-limit'd (selbst seltener möglich, aber bei Auth-Storms theoretisch), wird in einer Schleife re-versucht. Plus: gleichzeitige Re-Logins von 4 Coordinator-Instanzen → 4× rpcd-Auth-Storm in <1 s.

**Plan.**
- Exponential-Backoff zwischen Login-Fehlversuchen pro Host: 1 s → 2 s → 4 s → 8 s, max 60 s.
- Zustand `self._auth_backoff_until: float` existiert teilweise schon — sollte konsequenter genutzt werden.
- Globaler Lock bei parallelen Re-Logins auf denselben Host: `asyncio.Lock` in `_OpenWrtAPI`.
- **Test.** Mock 3 fehlschlagende `session/login` → Backoff steigt, max 60 s, dann Erfolg → Backoff resettet.

**Files.** `custom_components/openwrt_router/api.py`.

**Risiko.** Niedrig — defensive Logik, keine Verhaltensänderung im Happy-Path.

### 5. Configurable SCAN_INTERVAL via Options-Flow

**Problem.** `SCAN_INTERVAL_SECONDS` ist eine Konstante. User kann sie nicht ohne Code-Patch ändern. Nach dem Crash war die User-Empfehlung "test mit 120 s" — aber das bedeutet const.py-Patch auf Production, der bei HACS-Update verloren geht.

**Plan.**
- Neuer Options-Flow `OptionsFlowHandler` mit Schema `{"scan_interval": int (60–600)}`.
- `OpenWrtCoordinator.__init__` liest `entry.options.get("scan_interval", SCAN_INTERVAL_SECONDS)`.
- HA-UI: Integration → Configure → "Scan Interval (seconds)".
- Default bleibt 90 (siehe Punkt 3), Range 60–600.
- **Test.** Bestehender `mock_config_entry` + Options-Schema + Coordinator-Init verifiziert dass Override durchgereicht wird.

**Files.** `custom_components/openwrt_router/config_flow.py` (Options-Flow neu), `coordinator.py` (Init-Änderung), `strings.json` (UI-Labels).

**Risiko.** Niedrig — additive UI-Änderung, kein Migration nötig (default fallback auf const).

### 6. F5 Aggregator — bedingt aktivieren

**Bedingung.** F5 (`/root/ha-collect.sh` + `_call_aggregator()`) wird in v1.19 NUR aktiviert WENN beide Bedingungen erfüllt sind:

1. Punkt 1–3 sind umgesetzt UND auf Production validiert (Coordinator-Update-Dauer **konsistent** unter `SCAN_INTERVAL_SECONDS`).
2. Wenn Update-Dauer trotz 1–3 weiter > 30 s pro Router liegt — dann ist F5 die Lösung. Falls < 15 s — F5 ist nicht nötig.

**Plan (falls aktiviert).**
- `/root/ha-collect.sh` bash-Skript auf jedem Router deployen (manuell oder via `acl_provisioning.py`-Erweiterung).
- `OpenWrtAPI._call_aggregator()` Implementation füllen (heute `NotImplementedError`).
- Coordinator wählt zwischen ubus-Pfad (default) und Aggregator-Pfad via `entry.options["use_aggregator"]`.
- Schema-Version-Validation existiert bereits in v1.18.0 (`AGGREGATOR_SCHEMA_VERSION = "1.18.0"`).

**Plan (falls NICHT aktiviert).**
- F5-Skeleton bleibt im Repo, `NotImplementedError` als Wand bestehen.
- Reserve für v1.20+, falls Coordinator-Last weiter steigt.

**Files.** `api.py`, `coordinator.py`, `acl_provisioning.py`, NEU: `scripts/ha-collect.sh`.

**Risiko.** Hoch — neue Roundtrip-Surface, neuer Schema-Vertrag, Router-Side-Tooling-Update. Daher Bedingung.

## Out-of-Scope für v1.19

- ❌ Keine Refactors an Subprocess-Lifecycle (v1.18.0 ist gut).
- ❌ Keine Panel-Lifecycle-Änderungen (v1.18.0 löst das saubere Problem).
- ❌ Kein Async-SSH (asyncssh) — bleibt v1.20+ Roadmap.
- ❌ Kein Rewrite des Topology-Mesh-Aggregators.

## Akzeptanzkriterien für v1.19-Release

1. **Coordinator-Update-Dauer < SCAN_INTERVAL** in 99% der Polls auf Production-Hardware (gemessen über 24h).
2. **Re-Login-Rate < 0.1 / Sekunde** im Steady-State (vs. 4+ heute).
3. **Crash-freie 7 Tage** auf Production nach Deploy (vorher: sporadisch alle ~1–3 Tage).
4. **Tests grün** — bestehende 484 + neue Token-Refresh + Coordinator-Lock + Options-Flow Tests.
5. **Downgrade-Safe** — kein Schema-Change, der einen Rollback-auf-v1.18.0 verbieten würde.
